fmtlog源码扫读版本

核心优势

  • SPSC（单生产者单消费者）无锁队列
    • 每个线程都有自己的ThreadBuffer（内部维护一个SPSCVarQueue）
    • 无锁操作，纳秒级别的操作性能
  • 日志格式化异步化
    • 日志格式化是慢操作（fmt::format是CPU密集型）
    • fmtlog会把格式化操作放到后台线程中异步执行
    • 前端只做简单的参数编码和推入队列
  • 后端线程的最小堆调度机制
    • 后台线程从所有线程的队列中取日志时，使用一个最小堆维护每个线程当前队头的时间戳
    • 确保日志的全局时间顺序正确，且能保证一定程度的均匀调度
  • 自定义时间戳采样器
    • 使用RDTSC指令获取时间戳，比std::chrono快得多
    • 用一个轻量的时间校准机制（tscns）将tsc映射为纳秒，取代慢速系统调用
  • 刷盘策略合理
    • 写文件时不强制fflush，除非：
      • 手动调用poll(true)
      • flush超时到了
    • 避免I/O瓶颈拖慢写入线程
  • 内存预分配：内部使用fmt::MemoryBuffer的对象缓存，避免频繁分配和释放内存
  • 不拷贝写入：参数编码也不需要格式化字符串那样的对象生命周期控制

核心代码布局

  • fmtlog.h
    • 头文件只暴露接口和模板声明
  • fmtlog-inl.h
    • 包含所有实现细节，包括模板定义、inline函数实现等
  • fmtlog.cc
    • 通过#include "fmtlog-inl.h"把实现编译成一个.o文件，从而提供可选的非header-only使用方式

unNamedType用于从fmt::format相关的数据结构中提取实际类型Arg

函数模板isCstring和isString用于检查特定的模板参数Arg是否是特定类型的字符串

needCallDtor函数模板的作用是判断给定的模板参数Arg是否需要显式调用析构函数

encodeArgs用于将可变参数打包成一块连续内存out，供后续格式化或延迟写入时使用

storeNamedArgs实现了命名参数的存储功能，用于处理格式化字符串中的命名参数。其中，

  • Idx: 当前处理的参数索引
  • NamedIdx: 当前命名参数的索引，也即数组索引
  • named_args_store: 存储命名参数信息的数组

decodeArgs从一段连续的内存const char* in中，按格式把参数反序列化出来，恢复成fmt::basic_format_arg<Context>类型的数组args，供后续格式化输出使用

ValueOnly的作用

在fmt库中，格式化参数类型fmt::basic_format_arg<Context>本质上是个封装结构，内部有个union，叫做：

  • detail::value<Context>是一个union，代表具体的数据值
  • 在fmtlog中，为了节省开销、绕过封装，它直接用指针强转的方式访问了这个union

从data序列化数据中反序列化多个参数，填入args数组中，用于后续fmtlog格式化，并在必要时调用析构函数以管理资源

argIdx的作用

主要用作状态控制：

  • argIdx < 0: 表示第一次格式化这些参数，参数尚未反序列化过
  • argIdx >= 0: 表示这批参数已经反序列化过了，只需从data里恢复value，而不需要重新构造basic_format_arg

unNameFormat用于将命名参数格式化字符串（如{name}）转换成位置参数格式化字符串（如{0}、{1}），以便于后续更快格式化或进行参数重排。其中：

  • Reorder == false: 把{name}转换成{id}（数字编号）
  • Reorder == true: 只提取每个命名参数出现顺序，填入reorderIdx映射表，会把{name}转换成{}

参数：

  • logId: 日志格式在系统中的唯一ID，初始为0
  • tsc: 高精度时间戳
  • location: 日志位置（如main.cpp:100）
  • level: 日志级别
  • format: 格式字符串，带类型检查（fmt::format_string<>）
  • args…: 日志内容
