Boost::asio

Posted on 2025-03-31 Edited on 2025-04-18

知识点提炼

  • Boost.Asio使用io_context同操作系统的输入/输出服务进行交互
  • io_context是线程安全的，可以被多个线程同时调用
  • io_context::run()会阻塞线程，直到io_context内部的任务都完成
  • 单线程调用io_context::run(): 所有回调都在该线程执行
  • 多线程调用io_context::run(): 对个线程共享io_context的任务，并发执行回调
  • socket类不是线程安全的
  • close(): 关闭套接字后，调用这个套接字上任何的异步操作都会被立即关闭，同时返回error::operation_aborted错误码
  • shutdown(type_of_shutdown): 部分关闭套接字，是send或receive操作失效，或两者都失效。只是逻辑上的关闭，底层的TCP连接仍然保持
  • cancel(): 取消套接字上所有的异步操作。这个套接字上任何的异步操作都会立即结束，然后返回error::operation_aborted错误码。不会关闭套接字，可以继续使用
  • 套接字实例不能被拷贝，其拷贝构造函数和赋值运算符被删除或设为私有
  • 流可以随机访问，但是套接字是不可回溯的

run() 是 是 适用于长期运行的程序，如服务器

异步调用方法

方法 任务执行方式 是否立即执行 线程安全性 适用场景

wrap() 创建包装器，一般用于保证strand任务按顺序执行 取决于post()或dispatch() 是 保证任务顺序执行，即使任务在不同线程上运行
  • 为了使socket和缓冲区（read或者write）在整个异步操作的生命周期中一直活动，需要采取特殊的防护措施：连接类需要继承自enable_shared_from_this，然后在内部保存它需要的缓冲区，而且每次异步调用都要传递一个智能指针给this操作

C++异常处理笔记

Posted on 2025-03-07 Edited on 2025-04-11

知识点提炼

  • 异常事件发生时，如果一直回退到主函数main都不能处理该异常，则调用系统函数terminate终止程序
  • 执行throw语句时，其操作数的结果作为对象被复制构造为一个新的对象，放在内存的特殊位置：不是堆栈上
    • Windows上是放在线程信息块TIB中
    • Linux上是放在TCB(Thread Control Block)/TLS(Thread Local Storage)中

      在libstdc++（GCC的标准库实现）中，异常对象的存储和管理涉及__cxa_allocate_exception()和__cxa_get_globals()等函数。__cxa_allocate_exception()在堆上分配异常对象；GCC通过__cxa_get_globals()获取当前线程的异常处理数据结构__cxa_eh_globals，它通常存储在TLS或TCB中；__cxa_eh_globals结构体存储了当前线程的异常对象指针，但并不是存储异常对象本身

  • throw语句抛出的异常对象在匹配成功的catch语句的结束处被析构（即使该catch语句使用的是非引用的传值参数类型）
  • 由于throw语句都进行了一次副本拷贝，因此异常对象应该是可以复制构造的
    • C++98/03：异常对象必须支持拷贝构造
    • C++11及以后：可以支持移动构造，减少性能损耗
  • 抛出一个表达式时，被抛出对象的静态编译时类型将决定异常对象的类型
  • 异常规格在C++11中被弃用，C++11定义了新的noexcept关键字，如果在函数声明后直接加上noexcept关键字，表示函数不会抛出异常。如果保证不抛出异常的函数实际上抛出了异常，将直接调用terminate终止程序的运行

Linux页表体系笔记

Posted on 2025-02-21 Edited on 2025-04-11

虚拟内存与物理内存映射

  • 物理内存页：内核将整个物理空间划分为一页一页相同的内存块，每个内存块大小为4k（x86_64架构中默认配置）
  • struct page：内核中使用其来封装每页内存块的状态信息
  • PFN(Page Frame Number)：内核为每个物理内存页page统一编号，一一对应且全局唯一
  • 全局数组mem_map：表示系统中每一物理内存页，通常存储在内核的虚拟地址空间中；在内核的初始化阶段通过物理内存区域和页表映射来创建的
  • NUMA(Non-Uniform Memory Access)内存架构：与物理CPU(或CPU Socket)一一对应，每个物理CPU内的所有物理核心和逻辑核心都共享该节点的本地内存；每个NUMA节点的物理内存会被映射到全局mem_map中的一段区域
  • 进程虚拟内存空间分类
    • 未分配页面：非常庞大，远远超过物理内存空间
    • 已分配未映射页面：通过glibc的malloc接口或者系统调用mmap向内核申请虚拟内存
    • 正常页面：进程读写已分配未映射的虚拟内存页时，CPU中地址翻译的硬件MMU会产生缺页中断，随后内核分配对应的物理内存页面，并将虚拟内存页和物理内存页映射起来
  • 页表：存储进程虚拟内存与物理内存之间的映射关系，本质也是一个物理内存页；每一条数据称之为页表项PTE(Page Table Entry)

    在32位系统中页表中的PTE占用4个字节，64位系统中页表的PTE占用8个字节

  • struct mm_struct：描述进程的虚拟内存空间，进程间独立，且每个进程都有独立的页表；页表起始地址存放在struct mm_struct的pgd属性中，进程被创建出来时设置

    pgd刚被创建出无法被CPU直接使用，因为其还只是顶级页表的虚拟内存地址

  • 四级页表
    • PGD(Page Global Directory)：全局页目录
    • PUD(Page Upper Directory)：上层页目录
    • PMD(Page Middle Directory)：中间页目录
    • PT(Page Table)：页表
  • 虚拟内存地址的格式：页表内的偏移 + 物理内存页内偏移；结合页表起始地址可获取虚拟页在页表中对应的PTE
  • 进程调度：内核调用context_switch进行进程的上下文切换，主要包括：
    • 进程虚拟内存空间的切换
    • 寄存器以及进程栈的切换

      其中，进程虚拟内存空间的切换包括调用load_new_mm_cr3将进程顶级页表起始地址mm_struct->pgd中的虚拟内存地址通过__sme_pa宏转换为物理内存地址，并将pgd的物理内存地址加载到cr3寄存器中

  • 用户进程正常的内存访问是在用户态进行的，无需进入内核态，除非访问的是以下虚拟内存页面类型：
    • 未分配页面
    • 已分配未映射页面
    • 已映射，但该虚拟内存页映射的物理内存页被置换到磁盘上了
  • 进程页表分为两个部分：
    • 进程用户态页表
    • 内核页表的copy部分
  • 内核线程之间的调度由于不涉及地址空间切换，因此不会引发TLB缓存和CPU高速缓存的刷新操作，从而避免了进程调度时的性能开销
  • 进程调度则涉及地址空间的切换，需要刷新TLB和CPU缓存，带来额外的性能开销
  • 页表本质上还是一个物理内存页，里面存放的是PTE，保存虚拟内存与物理内存的映射关系；会参与内核的调度，即，会被内核swap in和swap out，也会被缓存在CPU高速缓存中加速访问
  • MMU(Memory Management Unit)硬件加速地址翻译过程
  • CPU高速缓存L1、L2、L3 cache缓存热点PDE、PTE
  • MMU引入硬件缓存TLB(Translation Lookaside Buffer)缓存被MMU翻译之后的热点PTE

[WIP] 性能调优的技术

Posted on 2025-02-07 Edited on 2025-02-14

本文主要记录作者在工作过程中关于性能调优方面的点滴，仅涵盖C++

方法论

性能调优的流程

工具总结

CPU: top, vmstat, pidstat, sar, perf
内存: top, free, vmstat, cachetop, cachestat, sar, gperftools
磁盘: top, iostat, iotop, vmstat, pidstat, du/df
网络: netstat, sar, dstat, tcpdump
应用: profiler, dump分析

CPU

CPU利用率高本身不是问题，关注CPU负载更能判断系统计算资源是否健康

CPU利用率高 且 平均负载高

常见于CPU计算密集型应用，大量消耗CPU资源的应用场景有：

  1. 正则操作
  2. 数学运算
  3. 序列化/反序列化
  4. 反射操作
  5. 死循环或不合理的大量循环
  6. 基础/第三方组件缺陷

排查以及验证思路：通过Profiling的方式，得到应用一段时间内的on-CPU火焰图，进而定位问题。另外，CPU利用率过高也有可能是CPU本身性能瓶颈导致的，可进一步通过vmstat查看详细的CPU利用率。用户态CPU利用率（us）较高，说明用户态进程占用了较多的 CPU，如果这个值长期大于50%，应该着重排查应用本身的性能问题。内核态CPU利用率（sy）较高，说明内核态占用了较多的CPU，所以应该着重排查内核线程或者系统调用的性能问题。如果us + sy的值大于80%，说明CPU可能不足

CPU利用率低 且 平均负载高

常见于IO密集型应用

排查以及验证思路：使用vmstat 1定时输出系统资源使用，观察%wa(iowait)列的值，该列标识了磁盘IO等待时间在CPU时间片中的百分比，如果这个值超过30%，说明磁盘IO等待严重，这可能是大量的磁盘随机访问或直接的磁盘访问（没有使用系统缓存）造成的，也可能磁盘本身存在瓶颈，可以结合iostat或dstat的输出加以验证，如%wa(iowait)升高同时观察到磁盘的读请求很大，说明可能是磁盘读导致的问题。另外，耗时较长的网络请求（即网络IO）也会导致CPU平均负载升高，如MySQL慢查询、使用RPC接口获取接口数据等。这种情况的排查一般需要结合应用本身的上下游依赖关系以及中间件埋点的trace日志，进行综合分析

CPU上下文切换次数变高

排查以及验证思路：先用vmstat查看系统的上下文切换次数，然后通过pidstat观察进程的自愿上下文切换（cswch）和非自愿上下文切换（nvcswch）情况。自愿上下文切换，是因为应用内部线程状态发生转换所致，譬如调用sleep()、join()、wait()等方法，或使用了mutex锁结构；非自愿上下文切换，是因为线程由于被分配的时间片用完或由于执行优先级被调度器调度所致

如果自愿上下文切换次数较高，意味着CPU存在资源获取等待，比如说，IO、内存等系统资源不足等。如果是非自愿上下文切换次数较高，可能的原因是应用内线程数过多，导致CPU时间片竞争激烈，频频被系统强制调度

内存

内存分为系统内存和进程内存，一般遇到的内存问题，绝大多数都会落在进程内存上，系统资源造成的瓶颈占比较小

系统内存不足

排查以及验证思路：

  • 使用free查看当前内存的可用空间大小，然后vmstat查看具体的内存使用情况及内存增长趋势，这个阶段一般能定位占用内存最多的进程
  • 分析缓存/缓冲区的内存使用。如果这个数值在一段时间内变化不大，可以忽略。如果观察到缓存/缓冲区的大小在持续升高，则可以使用pcstat、cachetop、slabtop等工具，分析缓存/缓冲区的具体占用
  • 排除掉缓存/缓冲区对系统内存的影响后，如果发现内存还在不断增长，说明很有可能存在内存泄漏

内存泄漏

排查以及验证思路：

  • 查看进程的内存映射，如果堆区不断增长，可能有内存泄漏
  • smem -p显示进程的内存占用情况
  • 借助内存检测工具辅助定位，譬如valgrind/asan/gperftools

磁盘IO和网络IO

磁盘IO

排查以及验证思路：

  • 使用工具输出磁盘相关的输出的指标，常用的有%wa（iowait）、%util，根据输判断磁盘IO是否存在异常，譬如%util这个指标较高，说明有较重的IO行为
  • 使用pidstat定位到具体进程，关注下读或写的数据大小和速率
  • 使用lsof + 进程号，可查看该异常进程打开的文件列表（含目录、块设备、动态库、网络套接字等），结合业务代码，一般可定位到IO的来源，如果需要具体分析，还可以使用perf等工具进行trace定位IO源头

需要注意的是，%wa（iowait）的升高不代表一定意味着磁盘IO存在瓶颈，这是数值代表CPU上IO操作的时间占用的百分比，如果应用进程的在这段时间内的主要活动就是IO，那么也是正常的

网络IO

排查以及验证思路：

  • 一次传输的对象过大，可能会导致请求响应慢
  • 网络IO模型选择不合理，导致应用整体QPS较低，响应时间长
  • RPC 调用的线程池设置不合理
  • RPC 调用超时时间设置不合理，造成请求失败较多

  1. 显示系统整体的CPU利用率和闲置率

如果实参与引用参数不匹配，C++将生成临时变量。现在，仅当参数为const引用时，C++才允许这样做，编译器将在下面两种情况下生成临时变量：

  • 实参的类型正确，但不是左值。左值参数是可被引用的数据对象，例如，变量、数组元素、结构成员、引用和被解除引用的指针都是左值。非左值包括字面常量和包含多项的表达式。

this指针

每个成员函数（包括构造函数和析构函数）都有一个this指针，this指针指向调用对象。如果方法需要引用整个调用对象，则可以使用表达式*this。在函数的括号后面使用const限定符将this限定为const，这样将不能使用this来修改对象的值。

使用类

友元是否有悖于OOP

📝 因为友元机制允许非成员函数访问私有数据，很多人会认为友元违反了OOP数据隐藏的原则。不过，这个观点太片面了。相反，应将友元函数看作类的扩展接口的组成部分。例如，从概念上看，double乘以Time和Time乘以double是完全相同的。也就是说，前一个要求有友元函数，后一个使用成员函数，这是C++句法的结果，而不是概念上的差别。通过使用友元函数和类方法，可以用同一个用户接口表达这两种操作。另外，只有类声明可以决定哪一个函数是友元，因此类声明仍然控制了哪些函数可以访问私有数据。简而言之，类方法和友元只是表达类接口的两种不同机制。

转换函数

⚠️ 应谨慎的使用隐式转换函数。通常，最好选择仅在被显式地调用时才会执行的函数。

总之，C++为类提供了下面的类型转换：

  • 只有一个参数的类构造函数用于将类型与该参数相同的值转换为类类型。例如，将int值赋给Stonewt对象时，接受int参数的Stonewt类构造函数将自动被调用。不过，在构造函数声明中使用explicit可防止隐式转换，而只允许显式转换。
  • 被称为转换函数的特殊类成员操作符函数，用于将类对象转换为其他类型。转换函数是类成员，没有返回类型、没有参数、名为operator typeName()，其中，typeName是对象将被转换成的类型。将类对象赋给typeName变量或者强制转换为typeName类型时，该转换函数将自动被调用。

类继承

派生类构造函数要点

  • 基类对象首先被创建。
  • 派生类构造函数应通过成员初始化列表将基类信息传递给基类构造函数。
  • 派生类构造函数应初始化派生类新增的数据成员。
  • 释放对象的顺序与创建对象的顺序相反，即首先执行派生类的析构函数，然后自动调用基类的析构函数。

虚函数使用成本

  • 对每个类，编译器都创建一个虚函数表（数组）（virtual function table，vtbl）。
  • 每个对象都将增大，增大量为存储地址的空间，每个对象都会有一个虚表指针（vptr），其指向该类的虚函数表。
  • 每个函数调用都需要执行一步额外的操作，即到表中查找地址。
  • 非虚函数的效率比虚函数稍高，但不具备动态联编功能。

🖊️ 通常应给基类提供一个虚拟析构函数，即使它并不需要析构函数。
