fmtlog源码扫读版本

核心特点

  • SPSC（单生产者单消费者）无锁队列
    • 每个线程都有自己的ThreadBuffer（内部维护一个SPSCVarQueue）
    • 无锁操作，纳秒级别的操作性能
  • 日志格式化异步化
    • 日志格式化是慢操作（fmt::format是CPU密集型）
    • fmtlog会把格式化操作放到后台线程中异步执行
    • 前端只做简单的参数编码和推入队列
  • 后端线程的最小堆调度机制
    • 后台线程从所有线程的队列中取日志时，使用一个最小堆维护每个线程当前队头的时间戳
    • 确保日志的全局时间顺序正确，且能保证一定程度的均匀调度
  • 自定义时间戳采样器
    • 使用RDTSC指令获取时间戳，比std::chrono快得多
    • 用一个轻量的时间校准机制（tscns）将tsc映射为纳秒，取代慢速系统调用
  • 刷盘策略合理
    • 写文件时不强制fflush，除非：
      • 手动调用poll(true)
      • flush超时到了
    • 避免I/O瓶颈拖慢写入线程
  • 内存预分配：内部使用fmt::MemoryBuffer的对象缓存，避免频繁分配和释放内存
  • 不拷贝写入：参数编码也不需要格式化字符串那样的对象生命周期控制
  • 如果没有合适的回调处理，在队列满的场景下会丢日志

核心代码布局

  • fmtlog.h
    • 头文件只暴露接口和模板声明
  • fmtlog-inl.h
    • 包含所有实现细节，包括模板定义、inline函数实现等
  • fmtlog.cc
    • 通过#include "fmtlog-inl.h"把实现编译成一个.o文件，从而提供可选的非header-only使用方式

Boost::asio

Posted on 2025-03-31 Edited on 2025-04-18

知识点提炼

  • Boost.Asio使用io_context同操作系统的输入/输出服务进行交互
  • io_context是线程安全的，可以被多个线程同时调用
  • io_context::run()会阻塞线程，直到io_context内部的任务都完成
  • 单线程调用io_context::run(): 所有回调都在该线程执行
  • 多线程调用io_context::run(): 对个线程共享io_context的任务，并发执行回调
  • socket类不是线程安全的
  • close(): 关闭套接字后，调用这个套接字上任何的异步操作都会被立即关闭，同时返回error::operation_aborted错误码
  • shutdown(type_of_shutdown): 部分关闭套接字，是send或receive操作失效，或两者都失效。只是逻辑上的关闭，底层的TCP连接仍然保持
  • cancel(): 取消套接字上所有的异步操作。这个套接字上任何的异步操作都会立即结束，然后返回error::operation_aborted错误码。不会关闭套接字，可以继续使用
  • 套接字实例不能被拷贝，其拷贝构造函数和赋值运算符被删除或设为私有
  • 流可以随机访问，但是套接字是不可回溯的

监听循环

方法 阻塞 处理所有任务 适用场景
run() 是 是 适用于长期运行的程序，如服务器
poll() 否 否（仅执行当前可用任务） 适用于事件轮询，避免阻塞线程

异步调用方法

方法 任务执行方式 是否立即执行 线程安全性 适用场景
post() 总是异步执行 否 是 始终异步提交任务，不管哪个线程调用，都加入io_context队列
dispatch() 当前线程是io_context线程：同步执行 否则：异步提交到io_context 可能立即执行 是 优先在当前线程中执行，如果不在io_context线程，则变为异步提交
wrap() 创建包装器，一般用于保证strand任务按顺序执行 取决于post()或dispatch() 是 保证任务顺序执行，即使任务在不同线程上运行
  • 为了使socket和缓冲区（read或者write）在整个异步操作的生命周期中一直活动，需要采取特殊的防护措施：连接类需要继承自enable_shared_from_this，然后在内部保存它需要的缓冲区，而且每次异步调用都要传递一个智能指针给this操作

    协程支持

方式 适用情况 优势 劣势

踩过的坑

1. async_write本身是个组合操作，内部实际上是调用多次更底层的async_write_some()来把整个buffer写完。所以即使使用strand保证了所有handler是串行执行的，但是多个async_write同时发起时，它们的多个内部async_write_some可能会交错进行（因为同时调了多个async_write） 最佳实践

C++异常处理笔记

Posted on 2025-03-07 Edited on 2025-04-11

知识点提炼

  • 异常事件发生时，如果一直回退到主函数main都不能处理该异常，则调用系统函数terminate终止程序
  • 执行throw语句时，其操作数的结果作为对象被复制构造为一个新的对象，放在内存的特殊位置：不是堆栈上
    • Windows上是放在线程信息块TIB中
    • Linux上是放在TCB(Thread Control Block)/TLS(Thread Local Storage)中

      在libstdc++（GCC的标准库实现）中，异常对象的存储和管理涉及__cxa_allocate_exception()和__cxa_get_globals()等函数。__cxa_allocate_exception()在堆上分配异常对象；GCC通过__cxa_get_globals()获取当前线程的异常处理数据结构__cxa_eh_globals，它通常存储在TLS或TCB中；__cxa_eh_globals结构体存储了当前线程的异常对象指针，但并不是存储异常对象本身

  • throw语句抛出的异常对象在匹配成功的catch语句的结束处被析构（即使该catch语句使用的是非引用的传值参数类型）
  • 由于throw语句都进行了一次副本拷贝，因此异常对象应该是可以复制构造的
    • C++98/03：异常对象必须支持拷贝构造
    • C++11及以后：可以支持移动构造，减少性能损耗
  • 抛出一个表达式时，被抛出对象的静态编译时类型将决定异常对象的类型
  • 异常规格在C++11中被弃用，C++11定义了新的noexcept关键字，如果在函数声明后直接加上noexcept关键字，表示函数不会抛出异常。如果保证不抛出异常的函数实际上抛出了异常，将直接调用terminate终止程序的运行

Linux页表体系笔记

Posted on 2025-02-21 Edited on 2025-04-11

虚拟内存与物理内存映射

  • 物理内存页：内核将整个物理空间划分为一页一页相同的内存块，每个内存块大小为4k（x86_64架构中默认配置）
  • struct page：内核中使用其来封装每页内存块的状态信息
  • PFN(Page Frame Number)：内核为每个物理内存页page统一编号，一一对应且全局唯一
  • 全局数组mem_map：表示系统中每一物理内存页，通常存储在内核的虚拟地址空间中；在内核的初始化阶段通过物理内存区域和页表映射来创建的
  • NUMA(Non-Uniform Memory Access)内存架构：与物理CPU(或CPU Socket)一一对应，每个物理CPU内的所有物理核心和逻辑核心都共享该节点的本地内存；每个NUMA节点的物理内存会被映射到全局mem_map中的一段区域
  • 进程虚拟内存空间分类
    • 未分配页面：非常庞大，远远超过物理内存空间
    • 已分配未映射页面：通过glibc的malloc接口或者系统调用mmap向内核申请虚拟内存
    • 正常页面：进程读写已分配未映射的虚拟内存页时，CPU中地址翻译的硬件MMU会产生缺页中断，随后内核分配对应的物理内存页面，并将虚拟内存页和物理内存页映射起来
  • 页表：存储进程虚拟内存与物理内存之间的映射关系，本质也是一个物理内存页；每一条数据称之为页表项PTE(Page Table Entry)

    在32位系统中页表中的PTE占用4个字节，64位系统中页表的PTE占用8个字节

  • struct mm_struct：描述进程的虚拟内存空间，进程间独立，且每个进程都有独立的页表；页表起始地址存放在struct mm_struct的pgd属性中，进程被创建出来时设置

    pgd刚被创建出无法被CPU直接使用，因为其还只是顶级页表的虚拟内存地址

  • 四级页表
    • PGD(Page Global Directory)：全局页目录
    • PUD(Page Upper Directory)：上层页目录
    • PMD(Page Middle Directory)：中间页目录
    • PT(Page Table)：页表
  • 虚拟内存地址的格式：页表内的偏移 + 物理内存页内偏移；结合页表起始地址可获取虚拟页在页表中对应的PTE
  • 进程调度：内核调用context_switch进行进程的上下文切换，主要包括：
    • 进程虚拟内存空间的切换
    • 寄存器以及进程栈的切换

      其中，进程虚拟内存空间的切换包括调用load_new_mm_cr3将进程顶级页表起始地址mm_struct->pgd中的虚拟内存地址通过__sme_pa宏转换为物理内存地址，并将pgd的物理内存地址加载到cr3寄存器中

  • 用户进程正常的内存访问是在用户态进行的，无需进入内核态，除非访问的是以下虚拟内存页面类型：
    • 未分配页面
    • 已分配未映射页面
    • 已映射，但该虚拟内存页映射的物理内存页被置换到磁盘上了
  • 进程页表分为两个部分：
    • 进程用户态页表
    • 内核页表的copy部分
  • 内核线程之间的调度由于不涉及地址空间切换，因此不会引发TLB缓存和CPU高速缓存的刷新操作，从而避免了进程调度时的性能开销
  • 进程调度则涉及地址空间的切换，需要刷新TLB和CPU缓存，带来额外的性能开销
  • 页表本质上还是一个物理内存页，里面存放的是PTE，保存虚拟内存与物理内存的映射关系；会参与内核的调度，即，会被内核swap in和swap out，也会被缓存在CPU高速缓存中加速访问
  • MMU(Memory Management Unit)硬件加速地址翻译过程
  • CPU高速缓存L1、L2、L3 cache缓存热点PDE、PTE
  • MMU引入硬件缓存TLB(Translation Lookaside Buffer)缓存被MMU翻译之后的热点PTE
